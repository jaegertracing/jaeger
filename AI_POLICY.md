# AI Usage Policy for Jaeger Contributions

This document outlines guidelines for using AI-assisted tools when contributing to the Jaeger project.

## Purpose

As AI coding assistants become common in software development, we want to provide clear guidance to contributors. This policy ensures quality, transparency, and compliance while welcoming the productivity benefits of AI tools.

## Guiding Principle

> Development and review of code generated by AI tools should be treated no differently than human-written code.
> — [Linux Foundation Generative AI Policy](https://www.linuxfoundation.org/legal/generative-ai)

All contributions, regardless of how they were created, must meet Jaeger's quality standards and pass the same review process.

## Acceptable Use

The following AI-assisted activities are welcome and do not require special disclosure:

| Use Case | Examples |
|----------|----------|
| **Code completion** | GitHub Copilot, Cursor, Codeium suggestions |
| **Test generation** | Generating unit tests, test cases |
| **Documentation** | Drafting docs, comments, README updates |
| **Boilerplate code** | Scaffolding, repetitive patterns |
| **Refactoring** | Code cleanup suggestions |
| **Learning** | Understanding codebase, exploring solutions |

## Disclosure Guidelines

Please note in your PR description when:

- AI tools generated a **substantial portion** (>50%) of the code changes
- AI assisted with **architectural or design decisions**
- You used AI for **bulk code generation** across multiple files

A simple note like *"Used Copilot for test generation"* or *"Claude helped draft initial implementation"* is sufficient.

## Requirements

All AI-assisted contributions must still:

1. **Include DCO sign-off** — AI assistance does not change this requirement
2. **Pass code review** — Same standards apply to all code
3. **Include appropriate tests** — AI-generated code needs tests too
4. **Follow coding standards** — Must match Jaeger's style and patterns
5. **Be understood by the contributor** — You must be able to explain and maintain the code you submit

## What to Avoid

| Practice | Why |
|----------|-----|
| Submitting code you don't understand | You're responsible for maintaining it |
| Blindly accepting AI suggestions | May contain bugs, security issues, or outdated patterns |
| Using AI to bypass CI/tests | Defeats our quality gates |
| Ignoring license concerns | AI may suggest code similar to copyleft-licensed sources |

## License Compliance

When using AI tools, ensure:

- The AI tool's terms don't conflict with [Apache 2.0 license](https://www.apache.org/licenses/LICENSE-2.0)
- No copyrighted third-party code is included without proper licensing
- If the AI tool flags potential matches with training data, review the licensing

Most major AI coding tools (GitHub Copilot, Claude, etc.) have settings to filter out suggestions matching public code. Consider enabling these features.

## For GSoC and New Contributors

We encourage learning with AI tools! Some tips:

- **Use AI to understand** — Ask it to explain code, not just write it
- **Start small** — Let AI help with tests and docs while you learn the codebase
- **Ask questions** — If unsure about AI-generated code, ask in your PR
- **Focus on learning** — The goal is to grow as a contributor, not just merge code

## For GenAI-Related Features

Contributors working on Jaeger's GenAI observability features ([#7827](https://github.com/jaegertracing/jaeger/issues/7827), [#7841](https://github.com/jaegertracing/jaeger/issues/7841)) should:

- Follow the same AI usage guidelines as other contributions
- Be familiar with [OpenTelemetry GenAI semantic conventions](https://opentelemetry.io/docs/specs/semconv/gen-ai/)
- Consider how AI tools might interact with the features being built
