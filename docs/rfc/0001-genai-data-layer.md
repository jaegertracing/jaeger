
**Technical Specification: Jaeger GenAI Observability Data Layer**

## **1. Objective**

Design an extensible data layer and schema within the existing Jaeger backend infrastructure to support GenAI observability. The goal is to manage evaluation (eval) results, benchmark datasets, and their correlations with distributed traces, without introducing external SQL-based databases or breaking Jaeger’s single-backend architecture.

## **2. Architectural Constraints**

* **Backend Consistency:** Utilize existing Jaeger storage backends (ClickHouse, OpenSearch, etc.). Do not introduce new database engines.  
* **Storage Paradigm:** Leverage the capability of current key-value/document backends (specifically ClickHouse) to act as analytical engines.  
* **Decoupled Entities:** Recognize that Evals, Datasets, and Traces have independent lifecycles.  
  * **Relationship Model:** Eval (1:N) Samples (1:N) Traces.  
* **Data Flexibility:** Schemas must support a "Core + Flexible" approach:  
  * **Core Columns:** Fixed metadata (IDs, Timestamps, References).  
  * **Customer-Defined Columns:** Arbitrary multi-column JSON structures for virtual judge scores, metrics, and dataset-specific sample attributes.

## **4. Design Requirements for the Agent**

Design the schema and access layer with the following considerations:

1. **Storage Engine Mapping:** Map these relational entities to the chosen backend (e.g., how to represent the EvalResult matrix in ClickHouse using Map types or sparse columns to avoid massive table alterations).  
2. **Indexing & Querying:** Propose indexing strategies for high-frequency search fields (e.g., EvalID, DatasetID, ModelID) while maintaining efficient retrieval of the arbitrary JSON metrics.  
3. **Schema Evolution:** Ensure the design allows users to add new "Virtual Judges" or custom metrics without requiring DDL migrations on the storage backend.  
4. **API/Query Interface:** Define how to perform analytical queries across these entities, such as:  
   * *Comparison:* "Compare Eval A vs. Eval B across Dataset X."  
   * *Drift Detection:* "Filter samples by a specific prompt version and aggregate faithfulness scores."  
   * *Trace Correlation:* "Retrieve all traces associated with samples where a specific metric score is below a threshold."

## **5. Success Criteria**

* **No Infrastructure Bloat:** Operates within the existing Jaeger storage cluster.  
* **Analytical Performance:** Supports sub-second aggregations for quality metrics across thousands of samples/traces.  
* **Interoperability:** Seamlessly links GenAI quality data with traditional distributed tracing data via TraceID references.
